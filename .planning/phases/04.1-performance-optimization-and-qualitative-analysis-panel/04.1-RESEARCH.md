## Summary

Phase 04.1 has two workstreams: backend performance optimization and a new qualitative analysis panel. The backend work involves three targeted changes to `supervisor.py` (batched tool calls, prompt caching, WebSocket token streaming) plus a new async Sonnet call for narrative polishing. The frontend work involves one new component (`QualitativeAnalysisPanel`) added below the existing two-column layout, two new Zustand state slices, and two new WebSocket event handlers.

The biggest technical risk is the **prompt caching minimum token requirement**. The Anthropic API enforces per-model minimums: Opus 4.6 requires **4096 tokens**, Sonnet 4.6 requires **2048 tokens**. Both current system prompts (~130-150 tokens) are far below these thresholds. Applying `cache_control` to short prompts silently does nothing — no error, no savings. The system prompts must be substantially padded with relevant context (behavioral baselines, rule registry state, episode history) to reach the minimum before caching will take effect.

The streaming implementation is straightforward: `client.messages.stream()` returns an `AsyncMessageStreamManager`, and `text_stream` is an `AsyncIterator[str]` that yields text deltas. The supervisor call can be converted from `messages.create()` to `messages.stream()` with a context manager, broadcasting each token chunk over WebSocket during iteration. The final complete message is recovered via `stream.get_final_message()`.

**Primary recommendation:** Implement batched tool calls first (highest latency impact), then streaming (highest demo impact). Prompt caching should only be added after padding system prompts to meet per-model minimums — otherwise it is dead code.

### Streaming API Pattern (HIGH confidence — verified from SDK source)

The `messages.stream()` method on `AsyncMessages` is the correct primitive:

```python
# Source: verified from AsyncMessageStream SDK source, anthropic 0.86.0
async with llm_client.messages.stream(
    model=models["supervisor"],
    max_tokens=1024,
    system=supervisor_prompt,
    messages=[{"role": "user", "content": user_message}],
) as stream:
    async for text_chunk in stream.text_stream:
        # text_chunk is a str delta
        await ws.broadcast("supervisor_token", episode_id, {"token": text_chunk})
    final_message = await stream.get_final_message()
```

`text_stream` is an `AsyncIterator[str]` that yields raw text deltas. `get_final_message()` returns the complete `Message` object after the stream finishes. The full message must be recovered this way to extract `supervisor_reasoning` for downstream use.

## Architecture Patterns

### Pattern 1: Batched Tool Calls (D-01)

**What:** Modify `PAYMENT_AGENT_SYSTEM_PROMPT` to instruct the Payment Agent to call all 3 tools in a single response turn. The existing `supervisor.py` turn-loop already handles multiple `tool_use` blocks per assistant message (the inner `for block in agent_response.content` loop). No structural change to the loop is needed — only the prompt changes.

**Current behavior:** The agent makes one tool call per turn (3+ turns for 3 tools).

**Target behavior:** Turn 1 → agent returns 3 `tool_use` blocks simultaneously. Turn 2 → all 3 `tool_result` blocks returned → agent produces `PaymentDecision` in `end_turn`.

**Prompt change location:** `sentinel/agents/payment_agent.py`, `PAYMENT_AGENT_SYSTEM_PROMPT` string.

**Key addition to prompt:**
```
When processing a payment request, call all required tools simultaneously in a single response.
Do not call tools one at a time — gather check_counterparty, verify_kyc, and read_invoice
(if an invoice is present) in one turn. This is faster and the results will all be returned
to you at once.
```

**Verification needed:** The prompt must be tested against a live API call to confirm the model complies. Claude's tool_choice defaults to "auto" — the model may choose not to batch. If needed, the prompt wording can be strengthened or `tool_choice: {"type": "any"}` can be added to force tool use (but this may conflict with the final `end_turn`).

**Risk:** Claude Sonnet 4.6 may not reliably batch all 3 tools in one call without strong prompt pressure. The `read_invoice` tool is conditional (only called if invoice_id is present in the payment request). Testing should verify both scenarios: with and without invoice.

### Pattern 2: Streaming Supervisor Tokens (D-03)

**What:** Convert the Supervisor reasoning call (`step 3` in `supervisor.py`) from `messages.create()` to `messages.stream()`. Broadcast each text delta as a `supervisor_token` WebSocket event. Recover the full response via `get_final_message()`.

**Integration point:** `supervisor.py` lines 117-128 (the `supervisor_response = await llm_client.messages.create(...)` block).

**New EventType values needed:**
- `"supervisor_token"` — streaming text delta from Supervisor reasoning
- `"narrative_template"` — immediate template-filled narrative (fires after `gate_evaluated`)
- `"narrative_ready"` — LLM-polished narrative (fires 3-5s later)

**EventType Literal** is in `sentinel/schemas/events.py`. Currently has 9 values. Add 3 more.

**Frontend handler:** `useWebSocket.js` switch statement needs cases for the 3 new events.

### Pattern 3: Hybrid Narrative Generation (D-09)

**What:** Two-phase narrative approach. Phase A (synchronous, instant): after `gate_evaluated` fires, build template-filled narrative strings from structured `VerdictBoard` data and broadcast as `narrative_template`. Phase B (async, 3-5s later): fire a Sonnet call to polish the 3 prose cards, broadcast result as `narrative_ready`.

**Template fill logic (Phase A):** Pure Python string interpolation from `VerdictBoard` fields. No LLM call. Executes in-process after `gate_result` is computed, before or alongside `episode_written`.

**Async polish call (Phase B):** `asyncio.create_task()` after broadcasting `gate_evaluated`. Does not block the main investigation path. Uses `models["agent"]` (Sonnet 4.6). Produces plain English prose for cards 1, 2, and 3. Card 4 (self-improvement arc) is always template-driven and never polished by LLM.

### Anti-Patterns to Avoid

- **String system prompt with cache_control:** The `system` parameter must be a list of blocks (`[{"type": "text", "text": "...", "cache_control": {...}}]`), not a plain string. Passing a string with top-level `cache_control` is valid but less precise — use block-level for control.
- **Blocking the gate on narrative:** The async polish call must be `asyncio.create_task()` (fire-and-forget), not `await`. Awaiting it would add 3-5s to the gate decision path.
- **Using `messages.stream()` with tools:** The stream context manager works with tools, but partial tool_use JSON accumulates via `InputJSONDelta` events — do not try to parse incremental tool JSON. Only `text_stream` gives clean text deltas. For tool-using turns (Payment Agent), continue using `messages.create()`.
- **Applying cache_control before padding prompts:** Adding `cache_control` to prompts below the minimum threshold wastes time and creates misleading code. Pad first, cache second.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Collecting full streamed response | Manual string concatenation of text_chunk | `stream.get_final_message()` | SDK method returns complete `Message` object including stop_reason, usage, all content blocks |
| Parsing narrative polish response | Custom JSON parser | Simple string split on `---SECTION---` delimiter or JSON with known keys | Sonnet will return structured response if prompted with explicit format; no need for regex extraction |
| Event type validation | Custom string check | Extend `EventType` Literal in `sentinel/schemas/events.py` | Pydantic validates on serialize — adding to Literal is the correct pattern established in Phase 1 |
| Card state management | Custom React context | Extend existing Zustand store | Pattern already established for `ruleStreaming`/`streamingBuffer` — identical pattern for `narrativeData`/`narrativePolishing` |

**Key insight:** The existing `supervisor.py` loop already handles multiple tool_use blocks per turn — the `for block in agent_response.content` inner loop collects all tool_use blocks before sending results back. The batching change is purely a prompt change, not a code structure change.

## Common Pitfalls

### Pitfall 1: Prompt Caching Silent Failure
**What goes wrong:** `cache_control` is added to system prompts, code appears to work, but cache write tokens appear in API responses and cache read tokens never accumulate. Caching has no effect.
**Why it happens:** Anthropic enforces per-model minimum token counts silently. Opus 4.6 requires 4096 tokens, Sonnet 4.6 requires 2048 tokens. The current Supervisor prompt is ~130 tokens and Payment Agent prompt is ~154 tokens.
**How to avoid:** Extend system prompts with substantial content (behavioral baseline tables, rule registry taxonomy, investigation guidelines, example verdict patterns) before adding `cache_control`. Verify cache effectiveness by checking `usage.cache_read_input_tokens > 0` in the response on the second call.
**Warning signs:** `usage.cache_creation_input_tokens` shows 0 on first call even though `cache_control` is set.

### Pitfall 2: Streaming Breaks Payment Agent Tool Loop
**What goes wrong:** `messages.stream()` is applied to Payment Agent tool-use turns. `InputJSONDelta` events accumulate partial JSON for tool inputs, which cannot be parsed incrementally.
**Why it happens:** Streaming emits tool input JSON character-by-character via `InputJSONDelta`. There is no clean `text_stream` for tool-use responses.
**How to avoid:** Only use `messages.stream()` for the Supervisor reasoning call (step 3 in `supervisor.py`), which is a pure text response. Keep Payment Agent calls as `messages.create()`.
**Warning signs:** `async for text_chunk in stream.text_stream` yields nothing on a tool-use turn.

### Pitfall 3: Narrative Polish Blocks Gate Decision
**What goes wrong:** The async Sonnet call for narrative polishing is awaited inline. The gate decision endpoint takes 8+ seconds instead of completing at ~5s.
**Why it happens:** Narrative LLM call is 3-5s. If awaited, it adds directly to the critical path.
**How to avoid:** Use `asyncio.create_task(_generate_narrative_polish(...))` immediately after broadcasting `gate_evaluated`. The task runs concurrently with episode write and any downstream processing.
**Warning signs:** HTTP response from `/api/investigate` arrives 8-10s into the investigation instead of at gate evaluation time.

### Pitfall 4: PaymentDecision Not Produced After Batched Tools
**What goes wrong:** Payment Agent successfully calls all 3 tools in turn 1, receives all results in turn 2, but then makes additional tool calls instead of producing the final JSON decision.
**Why it happens:** The agent may feel it needs to verify more or may not recognize that all verification is complete.
**How to avoid:** The system prompt must explicitly state that after receiving all tool results, the agent should immediately produce the PaymentDecision JSON. Consider also lowering `max_tokens` on the final turn or adding an explicit instruction in the second message: "You have all the information needed. Provide your final PaymentDecision now."
**Warning signs:** `steps_taken` list has more than 3 entries after the batching change.

### Pitfall 5: `resetInvestigation` Leaves Narrative State Stale
**What goes wrong:** A new investigation starts, `investigation_started` fires, the tree resets, but the qualitative panel still shows previous investigation's narrative text.
**Why it happens:** `resetInvestigation()` in `store.js` does not currently reset `narrativeData` or `narrativePolishing` because those slices don't exist yet.
**How to avoid:** When adding the new Zustand slices, add `narrativeData: {...initial}` and `narrativePolishing: {...initial}` to the `resetInvestigation` action explicitly. The WebSocket handler for `investigation_started` already calls `s.resetInvestigation()` — this is the correct trigger.
**Warning signs:** Card 1 shows text from the previous attack when a new investigation starts.

### Template Fill Function (Phase A — synchronous, no LLM)

```python
def build_narrative_template(
    payment_decision: PaymentDecision,
    verdict_board: VerdictBoard,
    verdicts: list[Verdict],
    gate_result: dict,
    rule_sources: list[dict],
) -> dict:
    """Build template-filled narrative strings from structured investigation data.

    Called synchronously after gate_evaluated — no LLM call, no latency.
    Returns dict with keys: attack_narrative, agent_reasoning, prediction_summary, self_improvement_arc.
    """
    # Attack narrative
    decision_word = "blocked" if gate_result["decision"] == "NO-GO" else "flagged"
    attack_narrative = (
        f"A payment of ${payment_decision.amount:,.2f} to {payment_decision.beneficiary} was intercepted. "
        f"The agent claimed {payment_decision.confidence:.0%} confidence. "
        f"The system {decision_word} this transaction: {gate_result['attribution']}"
    )

    # Agent reasoning (1 sentence per agent)
    agent_lines = []
    for v in verdicts:
        if v is None:
            continue
        flags = ", ".join(v.behavioral_flags) if v.behavioral_flags else "no anomalies"
        agent_lines.append(f"{v.agent_id.capitalize()}: {flags}.")
    agent_reasoning = " ".join(agent_lines)

    # Prediction vs actual
    mismatches = verdict_board.prediction_errors or {}
    predicted_z = mismatches.get("predicted_z_score", "unknown")
    actual_z = mismatches.get("summary_score", "unknown")
    prediction_summary = (
        f"The system expected a confidence z-score near baseline "
        f"but found {actual_z:.2f}. This divergence triggered rule generation."
        if isinstance(actual_z, float) else
        "Prediction data unavailable for this investigation."
    )

    # Self-improvement arc (always template-driven)
    if rule_sources:
        latest = rule_sources[-1]
        if latest.get("version", 1) > 1:
            arc = f"Rule #{latest['rule_id']} fired on the second attack and was refined into Rule #{latest['rule_id']}-v{latest['version']} with tighter thresholds."
        else:
            arc = f"After this attack, Sentinel generated Rule #{latest['rule_id']}. It is now active in the Safety Gate."
    else:
        arc = "No rules generated yet. Confirm an attack to trigger learning."

    return {
        "attack_narrative": attack_narrative,
        "agent_reasoning": agent_reasoning,
        "prediction_summary": prediction_summary,
        "self_improvement_arc": arc,
    }
```

1. **Will Sonnet 4.6 reliably batch all 3 tool calls in one turn?**

   - What's unclear: Whether the model will choose to emit all 3 without additional pressure. The `read_invoice` call requires an `invoice_id` which may not always be present.
   - Recommendation: Test the prompt change against a live API call with the Phase 1 attack fixture before wiring into the demo path. If batching is unreliable, consider adding `tool_choice: {"type": "any"}` on turn 1 with max_tokens limited to force tool use, and allow turn 2 to be `end_turn`.

2. **How much content to add for prompt caching to be effective?**
   - What we know: Opus 4.6 requires 4096 tokens; Sonnet 4.6 requires 2048 tokens. Current prompts are ~130-154 tokens.
   - What's unclear: Whether adding extensive behavioral baseline tables, risk taxonomy, and rule registry contents makes the prompts semantically better (probably yes for Supervisor) or just padding.
   - Recommendation: For the Supervisor prompt, inject the full behavioral baseline table, the hardcoded rule taxonomy, and a detailed investigation protocol. For the Payment Agent, add explicit tool usage guidelines, format requirements, and claim verification criteria. These are genuinely useful context additions that also satisfy the token minimum.

3. **Should `supervisor_token` events be buffered in Zustand or passed through to a display component?**
   - What we know: The existing `appendStreamingBuffer` / `streamingBuffer` pattern is used for rule streaming in `RuleSourcePanel`.
   - What's unclear: Whether the Supervisor token stream should share the same buffer or use a separate one (the rule streaming and supervisor streaming may overlap in time).
   - Recommendation: Add a separate `supervisorTokenBuffer` / `appendSupervisorToken` / `clearSupervisorBuffer` slice in Zustand — parallel to `ruleStreaming` pattern. Display in the investigation tree's Supervisor node tooltip or a dedicated area in the left panel. (This is Claude's discretion per the CONTEXT.md.)
