## Spacing Scale

Standard 8-point scale — consistent with all existing panel components.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (`gap-1`, `px-1`) |
| sm | 8px | Compact element spacing (`gap-2`, `px-2`, `py-2`) |
| md | 16px | Default element spacing (`p-4`, `gap-4`) |
| lg | 24px | Section padding (`space-y-3` between cards in row = 12px; panel internal top margin `mt-3`) |
| xl | 32px | Layout gaps (not used in qualitative row) |
| 2xl | 48px | Major section breaks (not used in qualitative row) |
| 3xl | 64px | Page-level spacing (not applicable) |

Exceptions:
- Panel internal padding: `p-3` (12px) is used by compact panels (RuleSourcePanel, AerospikeLatency). The qualitative panel cards use `p-4` (16px) to match GateDecisionPanel and AnomalyScoreBar — narrative text needs more breathing room than code/data displays.
- Horizontal gap between the 4 cards within the row: `gap-3` (12px) — consistent with existing `space-y-3` gap between right-column panels.
- Full-width row bottom padding: `pb-4` (16px) — matches right-column's `p-4` outer padding.

## Typography

Pulled directly from existing component patterns in codebase. No new sizes or weights introduced.

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Card section label | 11px (`text-[11px]`) | 600 (`font-semibold`) | 1.2 | Inter | Section label above each of the 4 cards (e.g. "Attack Narrative") |
| Body prose | 13px (`text-[13px]`) | 400 (default) | 1.6 | Inter | Plain English narrative text within each card — slightly looser line height than existing 12px mono data text, to aid reading speed for judges |
| Loading / polishing indicator | 11px (`text-[11px]`) | 400 (default) | 1.2 | Inter | Inline status text: "Polishing with AI..." shown while async Sonnet call is in flight |
| Mono label | 10px (`text-[10px]`) | 400 (default) | 1.2 | Roboto Mono | Rule IDs, episode IDs, version tags within the self-improvement arc card (e.g. `#001-v2`) |

Rationale for 13px body (vs. 12px used elsewhere): the qualitative panel is the plain-English reading layer for judges. 12px is appropriate for dense data tables and code; 13px gives the narrative text enough visual weight to be skimmable at presentation distance.

## Color

All color tokens already defined in `frontend/index.html` Tailwind config. No new tokens.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0d1117` (`bg-dark`) | Full-width row outer background — same page background |
| Secondary (30%) | `#161b22` (`surface`) | Card background for each of the 4 qualitative cards (`bg-surface`) |
| Accent (10%) | `#57abff` (`primary`) | Loading/polishing pulse dot; inline rule ID badges in self-improvement arc |
| Destructive | `#f85149` (`danger`) | Not used in this panel — no destructive actions |

Accent reserved specifically for:
1. The "polishing" indicator pulse dot (matches the WS connected dot pattern in the header)
2. Rule ID text references within the self-improvement arc card (e.g. `Rule #001`, `Rule #001-v2`) — to visually link to the blue `primary` color used for Python keywords in the RuleSourcePanel

Semantic colors in qualitative panel:
- Agent finding summaries that confirmed the attack: `text-danger` (`#f85149`) for the flagged finding phrase only (e.g. "hidden text detected" within the Forensics summary sentence)
- Agent finding summaries that were clean: `text-success` (`#3fb950`) for clean findings (e.g. "no anomaly" phrase)
- Prediction errors: `text-warning` (`#e3b341`) for the "expected X but found Y" span — consistent with the warning color used for deviation signals elsewhere

Border: `#30363d` (`border-muted`) — `border border-border-muted` on all cards, matching every other panel.

## Component Anatomy: QualitativeAnalysisPanel

### Layout

```
<App>
  <header />                          <!-- existing, untouched -->
  <div class="flex flex-1 overflow-hidden">  <!-- existing, untouched -->
    <InvestigationTree />             <!-- existing, left column -->
    <RightColumnStack />              <!-- existing, right column, 6 panels -->
  </div>
  <!-- NEW: full-width third row -->
  <div class="border-t border-border-muted bg-bg-dark px-4 py-3 shrink-0">
    <QualitativeAnalysisPanel />
  </div>
</App>
```

The new row sits below the two-column flex layout. It uses `shrink-0` so it does not compress the main content area. It is always visible without scrolling (fixed height, see Card Height below).

### Card Row Structure

```
<div class="grid grid-cols-4 gap-3">
  <AttackNarrativeCard />
  <AgentReasoningCard />
  <PredictionActualCard />
  <SelfImprovementArcCard />
</div>
```

Use `grid grid-cols-4` (not flexbox) so all 4 cards maintain equal width at all viewport sizes. The dashboard is a fixed-layout demo screen; responsive collapse is not required.

### Card Height

Fixed at `h-36` (144px). Narrative text uses `overflow-y-auto` inside the card body so long prose scrolls within the card without pushing other panels. This keeps the full-width row from affecting the height of the two-column layout above.

### Individual Card Pattern

Each of the 4 cards follows the same shell — identical to the existing panel pattern:

```jsx
<div className="bg-surface rounded-lg border border-border-muted p-4 flex flex-col gap-2 h-36">
  {/* Card label */}
  <div className="text-[11px] font-semibold uppercase tracking-wider text-text-muted shrink-0 flex items-center justify-between">
    <span>{LABEL}</span>
    {/* Polishing indicator — only visible while LLM response is in-flight */}
    {polishing && (
      <span className="flex items-center gap-1 text-[11px] text-text-muted font-normal normal-case tracking-normal">
        <span className="pulse-dot inline-block w-1.5 h-1.5 rounded-full bg-primary" />
        Polishing...
      </span>
    )}
  </div>
  {/* Card body — prose text */}
  <div className="text-[13px] text-text-main leading-relaxed overflow-y-auto flex-1">
    {content || <span className="text-text-muted">{EMPTY_STATE_COPY}</span>}
  </div>
</div>
```

## State Machine: Polishing Indicator

Each card has a `polishing` boolean in Zustand. Lifecycle:

| Event | State |
|-------|-------|
| `gate_evaluated` fires | Template content rendered, `polishing: true` for Cards 1, 2, 3 |
| Async Sonnet response arrives (new WS event `narrative_ready`) | LLM prose replaces template content, `polishing: false` |
| `rule_generated` or `rule_deployed` fires | Card 4 content updated in place; `polishing: false` for Card 4 (no LLM polishing for arc — it is always template-generated from structured data) |
| New investigation starts (`investigation_started`) | All 4 cards reset to empty state, `polishing: false` |

Transition between template and LLM-polished content: replace in place with no animation. A fade (`transition-opacity duration-300`) is acceptable but not required — plain replace is sufficient for demo.

## WebSocket Events (new)

| Event | Direction | Payload | Handler |
|-------|-----------|---------|---------|
| `narrative_template` | server → client | `{ attack_narrative, agent_reasoning, prediction_summary, self_improvement_arc }` | Set all 4 cards; set `polishing: true` for cards 1-3 |
| `narrative_ready` | server → client | `{ attack_narrative, agent_reasoning, prediction_summary }` | Replace cards 1-3 prose; set `polishing: false` |

Note: `narrative_template` fires synchronously when `gate_evaluated` completes (no LLM call needed — template fill from structured VerdictBoard data). `narrative_ready` fires 3-5s later after async Sonnet polishing. Both events must be added to the `EventType` Literal in `sentinel/schemas/` (Claude's discretion as noted in CONTEXT.md D-09).
