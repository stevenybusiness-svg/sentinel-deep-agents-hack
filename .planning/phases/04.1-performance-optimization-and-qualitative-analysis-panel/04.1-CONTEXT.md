# Phase 04.1: Performance Optimization and Qualitative Analysis Panel - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Two tightly coupled concerns inserted urgently between Phase 4 and Phase 5:

1. **Performance Optimization** — Reduce the investigation pipeline from ~30s to demo-viable (<10s target). The 30s is primarily the Payment Agent multi-turn loop (up to 10 sequential LLM calls). This cannot ship into the demo arc without optimization.

2. **Qualitative Analysis Panel** — A new full-width panel at the bottom of the dashboard that tells the attack story in plain English for judges. Shows attack narrative, per-agent reasoning summaries, prediction vs. actual summary, and self-improvement arc. Quantitative panels already exist — this adds the "what does it mean" layer.

Phase boundary: optimize the backend pipeline AND add the qualitative panel to the frontend. No new attack scenarios, no new agents, no changes to Safety Gate scoring logic.

</domain>

<decisions>
## Implementation Decisions

### Latency strategy
- **D-01:** Batch Payment Agent tool calls — prompt the Payment Agent to call all 3 tools (check_counterparty, verify_kyc, read_invoice) simultaneously in a single LLM response. Claude supports multiple tool_use blocks per message. Target: reduce from ~5-10 turns to 2 turns (turn 1: all 3 tools called; turn 2: results received + PaymentDecision produced).
- **D-02:** Add `cache_control: {"type": "ephemeral"}` to Supervisor and sub-agent system prompts. First call pays full latency; repeat demo runs (Attack 2, retries) are significantly faster.
- **D-03:** Stream WebSocket events earlier — broadcast Supervisor reasoning tokens live (as they arrive from the streaming API) to the frontend. This hides remaining latency by making the 30s feel active and responsive to judges watching the dashboard.

### Qualitative panel content
- **D-04:** Show all 4 sections in plain English, minimal jargon: attack narrative, per-agent reasoning summary, prediction vs. actual summary, self-improvement arc. The goal is for a non-technical judge to understand what happened in 15 seconds of reading.
- **D-05:** Attack narrative — 2-4 sentences: what attack was attempted, how the agent was manipulated, what gave it away. Written in plain English, not technical field names.
- **D-06:** Per-agent reasoning summary — 1-2 sentences per agent (Risk, Compliance, Forensics) summarizing their key finding. No raw JSON, no field names — natural language only.
- **D-07:** Prediction vs. actual summary — plain English narration of the prediction errors: "The system expected X but found Y, which triggered rule generation."
- **D-08:** Self-improvement arc — narrative of rule generation + evolution: "After this attack, the system generated Rule #001. It fired on the next attack. After Attack 2, it evolved into Rule #001-v2 with tighter thresholds." Updates as rules are generated/evolved.

### Narrative generation method
- **D-09:** Hybrid approach — template-fill immediately when `gate_evaluated` fires (panels populate instantly with structured data mapped to plain English sentences). Then fire an async Sonnet call post-decision to replace template content with richer, more natural prose. Frontend updates the qualitative panel when LLM response arrives (3-5s later). Gate decision is never blocked by narrative generation.

### Dashboard panel placement
- **D-10:** New full-width row at the bottom of the dashboard, spanning both columns, below the existing 2-column layout. Does not displace any existing panels or violate the 6-panel constraint on the right column. Implemented as a horizontal card row with 4 cards (one per content section) or a tabbed/accordion layout if horizontal space is tight.

### Claude's Discretion
- Exact prompt wording for batched tool calls (testing required to confirm Claude calls all 3 in one turn)
- Specific `cache_control` placement in message arrays (ephemeral caching has rules about min token counts)
- WebSocket event name for Supervisor streaming tokens and narrative LLM response
- Zustand state shape for qualitative panel data (template strings + LLM-polished strings)
- Transition UX between template state and LLM-polished state (fade, replace, or none)
- Whether the 4 qualitative cards are arranged horizontally or as a 2x2 grid
- Font size and card height for the qualitative row — must not push existing panels off-screen

</decisions>

<specifics>
## Specific Ideas

- Plain English is the priority for the qualitative panel — "a lot of technical nuance can take away from the power of the product if too much information is in one screen"
- The Payment Agent currently allows up to 10 turns — structurally it shouldn't be the biggest bottleneck; the fix is prompting for parallel tool calls, not just capping turn count
- Streaming is the right approach for hiding latency — makes the waiting feel like the system is working visibly rather than a black box delay
- Self-improvement arc section should update as the investigation progresses (shows Rule #001 generated, then fired, then evolved) — not a static snapshot

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `store.js` `decisionLog` state — already exists for decision logging; qualitative panel can read from this or a new `narrativeData` state key
- `store.js` `ruleStreaming` / `streamingBuffer` / `appendStreamingBuffer` — streaming pattern already established; Supervisor token streaming can follow this pattern
- `RuleSourcePanel.jsx` — already implements streaming display with CSS syntax highlighting; reference for how to handle async-filling content
- `ConnectionManager.broadcast()` — already handles all WebSocket events; just needs new event types

### Established Patterns
- `asyncio.TaskGroup` for parallelism — already used for sub-agents, Payment Agent tool call batching must not break this pattern
- `cache_control` support exists in Anthropic SDK — needs to be added to system prompt messages as `{"type": "ephemeral"}` on the last system message block
- WebSocket event taxonomy (EventType Literal in schemas) — adding new events requires extending the Literal type

### Integration Points
- `supervisor.py` step 4 (Payment Agent loop) — primary target for batched tool call prompt change
- `supervisor.py` step 3 (Supervisor LLM call) — add streaming + cache_control here
- `App.jsx` layout — add third row below existing 2-column grid
- New WebSocket events will need: (a) EventType extension in schemas, (b) broadcast call in supervisor.py, (c) handler in useWebSocket.js, (d) state update in store.js

</code_context>
